# System Architecture

في هذا المشروع قررنا بناء النظام اعتماداً على مفهوم:

Service Oriented Architecture (SOA)

أي أن النظام لا يكون ملفاً واحداً كبيراً، بل يتم تقسيمه إلى عدة خدمات، وكل خدمة تكون مسؤولة عن جزء واضح من المشروع.

اعتمدنا هذا التصميم لأن المشروع يطلب فصل المسؤوليات بين الخدمات، وأن يكون الكود منظماً وقابلاً للتطوير والاختبار.

## الفكرة العامة

المستخدم يكتب Query من الواجهة.

الطلب يمر إلى Gateway Service.

بعدها يتم التعامل مع الاستعلام حسب نوع البحث المختار.

النظام يستخدم الخدمات التالية:

- Corpus Service
- Preprocessing Service
- Indexing Service
- Retrieval Service
- Query Refinement Service
- Evaluation Service
- Gateway Service

## الخدمات

### Corpus Service

هذه الخدمة مسؤولة عن التعامل مع Dataset.

وظيفتها:

- تحميل Documents
- تحميل Queries
- تحميل Qrels
- إعطاء معلومات عامة عن Dataset

لا تقوم هذه الخدمة بعمليات preprocessing أو retrieval.

### Preprocessing Service

هذه الخدمة مسؤولة عن معالجة النصوص قبل استخدامها في البحث.

تطبق على Documents و Queries نفس خطوات المعالجة، مثل:

- Normalization
- Tokenization
- Stopword Removal
- Stemming أو Lemmatization

### Indexing Service

هذه الخدمة مسؤولة عن بناء الفهارس.

تشمل:

- Inverted Index
- TF-IDF Index
- BM25 Index
- Embedding Index

الفهارس يتم بناؤها وحفظها حتى لا نعيد بناءها في كل مرة.

### Retrieval Service

هذه الخدمة مسؤولة عن تنفيذ عملية البحث.

تشمل:

- TF-IDF Retrieval
- BM25 Retrieval
- Embedding Retrieval
- Hybrid Serial Retrieval
- Hybrid Parallel Retrieval

وتعيد النتائج مرتبة حسب درجة الصلة.

### Query Refinement Service

هذه الخدمة مسؤولة عن تحسين الاستعلام قبل البحث.

يمكن استخدامها لإضافة كلمات قريبة أو إعادة صياغة الاستعلام.

يجب أن يكون بالإمكان تشغيل البحث مع هذه الخدمة أو بدونها حتى نستطيع المقارنة في التقييم.

### Evaluation Service

هذه الخدمة مسؤولة عن تقييم نتائج البحث باستخدام المقاييس المطلوبة:

- MAP
- Recall
- Precision@10
- nDCG

كما سيتم استخدامها لمقارنة النتائج قبل وبعد تطبيق Query Refinement.

### Gateway Service

هذه الخدمة تكون نقطة الدخول الأساسية للنظام.

تستقبل الطلب من الواجهة، ثم توجهه إلى الخدمة المناسبة.

## طريقة التواصل بين الخدمات

سيتم استخدام REST API بين الخدمات باستخدام FastAPI.

اخترنا REST API لأنه واضح وسهل الاختبار ومناسب لمشروع طلابي، كما أنه يحقق فكرة التواصل بين الخدمات المطلوبة في SOA.

## لماذا هذا التصميم مناسب؟

هذا التصميم يساعدنا على:

- فصل المسؤوليات بين أجزاء المشروع
- جعل كل خدمة قابلة للاختبار بشكل مستقل
- تسهيل تقسيم العمل بين أعضاء الفريق
- تسهيل تعديل أي جزء لاحقاً بدون التأثير على باقي النظام
- حفظ الفهارس والنماذج في ملفات جاهزة لإعادة استخدامها أثناء المقابلة

## تدفق العمل داخل النظام

1. المستخدم يكتب Query من الواجهة
2. Gateway Service يستقبل الطلب
3. Query يمر على Preprocessing Service
4. Retrieval Service يستخدم الفهرس المناسب
5. يتم ترتيب النتائج حسب النموذج المختار
6. يتم عرض النتائج للمستخدم
7. Evaluation Service تستخدم Queries و Qrels لحساب جودة النظام

## ملاحظة

هذا التصميم مبدئي، ويمكن تعديله أثناء تنفيذ المشروع إذا احتجنا إلى تحسين بنية الخدمات أو طريقة التواصل بينها.
